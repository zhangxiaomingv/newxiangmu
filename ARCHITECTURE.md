# ZKONER 架构文档

---

## 一句话

**输入品牌名 → 同时问豆包和 DeepSeek → 对比两个回答 → 打分 → 给出优化建议**

不需要任何 AI 当裁判。豆包和 DeepSeek 互相验证，和官网 ground truth 做对比。

## 评分维度详解

### 1. 提及度 (20%)

```
检查回答是否识别品牌:
  - 回答长度 > 800字 → 90-100 分 (模型很了解)
  - 回答长度 300-800 → 60-90
  - 回答长度 50-300 → 30-60
  - 回答长度 < 50 → 5-30

额外:
  - 品牌名出现在回答开头 → +10
  - 回答是"抱歉我不了解" → 直接 0 分
```

### 2. 准确性 (30%)

```
从 ground truth 提取关键词列表:
  gt_keywords = {品牌名, 产品名1, 产品名2, 行业词, 特点词...}

从回答中提取名词/专有名词:
  answer_keywords = {...}

匹配度 = gt_keywords ∩ answer_keywords / gt_keywords

  - 匹配 > 80% → 80-100
  - 匹配 50-80% → 50-80
  - 匹配 < 50% → 10-50
```

### 3. 深度 (20%)

```
统计回答中"具体信息"的密度:
  - 数字: "2024年" "月销2万台" "1000亿"
  - 专有名词: "SU7" "澎湃OS" "宁德时代"
  - 时间: "成立于" "上市于"
  - 数据: "%" "亿" "万"

密度 = 具体信息词数 / 总词数

  - 密度 > 5% → 80-100
  - 密度 2-5% → 50-80
  - 密度 < 2% → 10-50
```

### 4. 差异化 (15%)

```
检查回答是否区分品牌:
  - 提到至少 1 个竞品名 → +30
  - 提到和竞品的区别 → +30
  - 提到独特优势 → +20
  - 用了"唯一" "首个" "独特" 等词 → +10
```

### 5. 认知盲区 (15%)

```
检查 AI 不知道什么:
  - 明确说"我不知道" → 基础 60 (诚实)
  - 用语模糊: "可能是" "也许是" "一般来说" → -10/个
  - 回答完全是通用行业描述(没有品牌具体信息) → 0-20
  - 无盲区信号 → 90
```

## Gap 检测 (缺口)

对比 ground truth 和模型回答，找出缺失:

```
ground truth 有的                模型回答里没有
─────────────────────────────────────────────
"智能驾驶是核心卖点"       →    没提到智能驾驶
"2024年3月发布SU7"       →    只说了"有车型"没具体名字
"换电网络覆盖全国"         →    豆包没提换电，DeepSeek提了

缺口列表:
  { category: "准确性", severity: "critical",
    description: "两个模型都无法说出具体产品名称" }
  { category: "深度", severity: "moderate",
    description: "豆包的描述过于笼统，全是行业通用词" }
```

## 技术实现

### 文件结构

```
backend/app/ai_engine/
├── base.py                 # 抽象基类 EngineInput/EngineOutput
├── registry.py             # 引擎注册表 + 优先级
├── prompts.py              # 5 个标准问题 + 评估规则
├── dual_model_engine.py    # 核心引擎（爬取→双模型查询→评估→报告）
├── claude_adapter.py       # Claude 适配器(旧)
├── metaso_adapter.py       # 启发式适配器(旧, fallback)
└── mock_adapter.py         # 演示数据适配器(旧)

backend/app/
├── crawler/                # httpx + BeautifulSoup 爬虫
├── scoring/                # 5 维评分引擎
├── storage/                # SQLite 存储
└── routers/analysis.py     # FastAPI 路由
```

### 引擎优先级

```
dual (豆包+DeepSeek) → metaso (启发式) → mock (演示)

auto 模式下自动选择第一个可用的:
  - 有 DOUBAO_API_KEY 或 DEEPSEEK_API_KEY → dual
  - 都没有 → metaso (纯规则，不需要任何 API key)
  - 开发和演示 → mock
```

### API Keys

```bash
# 豆包 (火山引擎)  获取: console.volcengine.com
DOUBAO_API_KEY=sk-xxx
DOUBAO_MODEL=doubao-pro-32k

# DeepSeek  获取: platform.deepseek.com
DEEPSEEK_API_KEY=sk-xxx
DEEPSEEK_MODEL=deepseek-chat

# 不需要 Claude key
```

## 为什么是豆包 + DeepSeek

| | 豆包 | DeepSeek |
|---|---|---|
| 用户 | 国内最大 C 端 AI 应用 | 开发者/技术决策者 |
| 覆盖面 | 普通消费者搜索场景 | B2B/技术品牌认知 |
| API | 火山引擎，OpenAI 兼容 | 原生 OpenAI 兼容 |

两个模型合在一起，覆盖了中国 AI 搜索的主流用户。
